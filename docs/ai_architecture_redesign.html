01

Current Architecture Weaknesses

CRITICAL

Monolithic 7,560-line single file

All logic — AI engine, parsers, charts, UI state, scoring, prim calculations — lives in one index.html. A bug in one area risks breaking everything. Impossible to unit test or maintain incrementally.

HIGH

Stateless AI — no memory between sessions

The current AI assistant sends fresh prompts with no knowledge of previous recommendations, outcomes, or learned patterns. Every session starts from zero. The engine can never improve.

HIGH

Deterministic projections, not probabilistic forecasting

Run-rate calculations (kalan_tl / remDays) are linear extrapolations with no confidence intervals, no momentum weighting, and no scenario modeling.

HIGH

No behavior profiling — all reps treated identically

The engine generates the same style of output regardless of whether a rep is aggressive vs conservative, end-of-period sprint type vs consistent performer. Strategy cannot be personalized.

MEDIUM

AI engine is a prompt wrapper, not a decision system

The strategy engine in _runEngineCore() generates templated HTML cards from deterministic if/else logic, then optionally calls Claude for narrative. There is no reasoning layer — just template filling.

MEDIUM

No scoring models — all rankings are single-metric

Brick priority is determined by sira <= 333 && mi < 90. Pharmacy priority is by total volume. No composite scoring incorporating multiple signals (MI trend, GI, visit frequency, competitor pressure, potential).

MEDIUM

No alert system — users must discover problems manually

There are no proactive alerts for premium threshold risk, product decline, brick weakness, or competitor acceleration. Users must navigate to each page and interpret the data themselves.

LOW

No root cause analysis — only surface metrics

When TL% drops, the system shows the number but does not explain which brick, which product, or which competitive dynamics caused it. The "why" is not answered.

AI Engine Redesign

The engine evolves from a prompt-fill system into a layered pipeline: compute → score → decide → explain → learn.

🧠

Behavior Profiler Module

Analyzes each rep's sales history across periods to build a behavioral fingerprint. Detects:

  • Pace score — week-over-week variance (high = sprint style)
  • Product focus ratio — Panocer vs portfolio spread
  • Consistency score — coefficient of variation across weeks
  • End-period multiplier — W8/W9 vs W1–W4 average ratio
  • Brick loyalty score — % of visits to same top-10 bricks
Output example
// Behavioral profile object
{
  ttt: "BUSE ÖZEN",
  style: "sprint_type",       // aggressive | conservative | sprint_type
  consistencyScore: 0.42,  // 0–1, lower = more consistent
  endPeriodMult: 1.87,     // W8+W9 / W1-W4 avg
  focusProduct: "PANOCER",  // dominant product
  brickLoyalty: 0.71,      // visits same bricks
  riskProfile: "high_variance"
}
🎯

Next Best Action Engine

Produces a single, ranked, actionable recommendation per rep per day. Not a list — one action, with justification and expected impact.

  • Input signals: gap-to-target, remaining days, brick scores, pharmacy pipeline, product weakness, rep behavior profile
  • Output: which brick, which pharmacy, which product, expected TL impact
  • Decision tree: if <85% and >10 days → focus on high-MI bricks; if 85–91% and <5 days → push PANOCER in top-10 eczane
Output shape
{
  action: "VISIT_BRICK",
  target: "SAMSUN MERKEZİ",
  product: "PANOCER",
  eczane: "BARIŞ ECZANESİ",
  expectedImpact_TL: 4200,
  confidence: 0.78,
  reasoning: "MI 87, below threshold..."
}
⚠️

AI Alert System

Proactive, priority-ranked alerts generated on each data load. Stored in a persistent alert log with timestamps and resolution status.

  • PREMIUM_RISK — TL% trajectory won't reach 91% by period end
  • PRODUCT_DECLINE — Week-over-week drop >15% for any product
  • BRICK_WEAKNESS — MI fell below 90 on a top-100 brick
  • COMPETITOR_SURGE — Competitor PPI grew >5pp in last 2 weeks
  • LOW_MOMENTUM — 3-week moving average declining
  • PORTFOLIO_IMBALANCE — Prim score drifting below 91%
🔍

Root Cause Analyzer

When a KPI drops, the analyzer walks back through the causal chain to produce a structured explanation.

  • Level 1: Which product caused overall TL% drop?
  • Level 2: Which brick(s) drove that product's decline?
  • Level 3: Is the cause internal (fewer visits) or external (competitor gaining market)?
  • Level 4: Which pharmacy cluster is the origin point?
RCA output
{
  trigger: "GENEL_TOPLAM dropped 4.2pp",
  cause: "ACIDPASS -8% in TEKKEKÖY brick",
  driver: "competitor_gain",  // or visit_gap
  evidence: [
    "Competitor PPI: +6pp in 2 weeks",
    "Own volume flat, market +12%"
  ]
}

Memory System Design

💾

Strategy Memory (localStorage)

Each AI recommendation is stored as a JSON record with a unique ID, timestamp, rep, context snapshot, and outcome flag. Max 500 entries with LRU eviction.

// Key: pharma_memory_v2
{
  id: "rec_20260115_BUSE_P1",
  ttt: "BUSE ÖZEN",
  period: "01/2026",
  recommendation: { action, product, brick },
  contextSnapshot: { tl_pct_at_time: 78 },
  outcome: "pending",  // success | fail
  outcomeValue: null   // actual TL gain
}
📊

Pattern Library (IndexedDB)

Aggregated patterns learned from outcomes. "PANOCER focus in sprint-type reps when TL 80–88% with 8+ remaining days shows 73% success rate." Used to weight future recommendations.

{
  patternId: "panocer_focus_sprint",
  conditions: {
    tl_pct_range: [80, 88],
    rep_style: "sprint_type",
    days_remaining_min: 8
  },
  successRate: 0.73,
  sampleSize: 11,
  avgImpact_TL: 8400
}
🔄

Outcome Tracker

At the start of each new data load, the system compares stored recommendations against the new TL% data to automatically classify outcomes as success/fail.

  • Load new CSV → compare current TL% vs TL% at recommendation time
  • If TL% improved >2pp after recommendation → mark as success
  • Update pattern library with new sample
  • Surface "what worked" summary in engine UI

Memory budget: A single recommendation record is ~400 bytes. 500 records = ~200KB, well within the 5MB localStorage limit. IndexedDB patterns can hold thousands of aggregated records. No backend required for Phase 1–2.

05

Forecasting Engine Design

📈

Momentum-Weighted Projection

Instead of remaining_TL / remaining_days, use an exponentially weighted moving average of recent week performance, adjusted by the rep's behavioral profile.

function calcMomentumRate(weeklyData, profile) {
  // EMA with alpha=0.35 (recent weeks weighted more)
  const alpha = 0.35;
  const ema = weeklyData.reduce((prev, w) =>
    alpha * w + (1 - alpha) * prev
  );
  // Adjust for sprint-type reps (they accelerate in W8-W9)
  const sprintBoost = profile.style === 'sprint_type'
    ? 1 + (profile.endPeriodMult - 1) * 0.4
    : 1.0;
  return ema * sprintBoost;
}
🎲

Probability Distribution

Use a normal distribution approximation (CLT) to estimate threshold probabilities from historical week variance. No external library needed.

function calcThresholdProbability(current, target, mu, sigma) {
  // Z-score for reaching threshold
  const z = (target - current - mu) / Math.max(sigma, 0.01);
  // 1 - Φ(z) via polynomial approximation (Abramowitz)
  return normalCDF(-z);
}

// Output: probability of reaching 85%, 91%, 100%
{
  p_85: 0.91,   // 91% chance of reaching 85%
  p_91: 0.67,   // 67% chance of reaching 91% (premium)
  p_100: 0.29,  // 29% chance of reaching 100%
  scenario_base: 88.4,
  scenario_bull: 94.1,
  scenario_bear: 82.7
}

Scoring Engine Design

All scores are normalized 0–100. Composite scores are weighted sums. No ML required — deterministic formulas with configurable weights.

Score ModelInputsFormulaUsed By
🧱 Brick Potential sira, MI, GI, MI trend, competitor PPI, visit gap days 0.3×(1-sira/500) + 0.25×MI/150 + 0.25×GI/150 + 0.2×(1-compPPI/100) Next Best Action, Daily Plan, Alert
💊 Pharmacy Potential total volume, product spread, recency of last order, brick rank 0.4×volume_norm + 0.3×spread_score + 0.3×recency_score Pharmacy priority lists, Route planning
👤 Rep Efficiency TL%, prim score, brick coverage rate, consistency score, trend direction 0.35×tl_pct/100 + 0.25×prim/100 + 0.2×coverage + 0.2×(1-variance) Manager dashboard, Team ranking

🚨 Risk Level days remaining, gap to 91%, declining MI bricks count, bear scenario TL% 0.4×gapUrgency + 0.3×miRiskCount/10 + 0.3×(1-p_91) Alert system, Color-coded UI indicators

📝

Learning Feedback Loop

After each period closes, the recommendation engine compares its predictions to actual outcomes and adjusts strategy weights. Implemented as a simple Bayesian update on pattern success rates.

function updatePatternWeight(patternId, success) {
  const p = memoryDB.getPattern(patternId);
  const n = p.sampleSize + 1;
  // Incremental Bayes: new_rate = (old_rate * n-1 + outcome) / n
  p.successRate = (p.successRate * (n-1) + (success ? 1 : 0)) / n;
  p.sampleSize = n;
  memoryDB.savePattern(p);
}

// Minimum 5 samples before a pattern influences decisions
// Patterns with successRate < 0.4 are deprioritized

ai/memory.js — Lightweight Memory Layer

// ai/memory.js — Pharmaceutical Sales AI Memory System
const MEMORY_KEY = 'pharma_ai_memory_v2';
const PATTERN_KEY = 'pharma_ai_patterns_v2';
const MAX_RECORDS = 500;

export const MemoryStore = {

  saveRecommendation(rec) {
    const records = this.getAll();
    const entry = {
      id: `rec_${Date.now()}_${rec.ttt.replace(/\s/g,'_')}`,
      ttt: rec.ttt,
      period: rec.period,
      strategy: rec.strategy,
      recommendation: rec.action,
      contextSnapshot: { tl_pct: rec.tl_pct, remDays: rec.remDays },
      outcome: 'pending',
      ts: Date.now()
    };
    records.push(entry);
    // LRU eviction if over limit
    if (records.length > MAX_RECORDS)
      records.splice(0, records.length - MAX_RECORDS);
    localStorage.setItem(MEMORY_KEY, JSON.stringify(records));
    return entry.id;
  },

  resolveOutcomes(currentData) {
    const records = this.getAll();
    let changed = false;
    records.forEach(r => {
      if (r.outcome !== 'pending') return;
      const current = currentData.find(d => d.ttt === r.ttt);
      if (!current) return;
      const delta = current.tl_pct - r.contextSnapshot.tl_pct;
      if (delta > 2)       { r.outcome = 'success'; r.delta = delta; changed = true; }
      else if (delta < -1) { r.outcome = 'fail';    r.delta = delta; changed = true; }
    });
    if (changed) localStorage.setItem(MEMORY_KEY, JSON.stringify(records));
  },

  getSuccessRate(strategy, repStyle) {
    const relevant = this.getAll()
      .filter(r => r.strategy === strategy &&
                   r.outcome !== 'pending');
    if (relevant.length < 5) return null; // not enough data
    return relevant.filter(r => r.outcome === 'success').length / relevant.length;
  },

  getAll() {
    try { return JSON.parse(localStorage.getItem(MEMORY_KEY) || '[]'); }
    catch { return []; }
  }
};

ai/forecaster.js — Probability Engine

// ai/forecaster.js — Probabilistic Forecasting Engine
import { normalCDF } from '../utils/math.js';

export function buildForecast(weeklyTL, target, remDays, profile) {
  const current = weeklyTL.reduce((s, v) => s + v, 0);
  const validWeeks = weeklyTL.filter(v => v > 0);

  // Weighted average: recent weeks count more
  const weights = validWeeks.map((_, i) => Math.pow(1.3, i));
  const wSum = weights.reduce((a, b) => a + b, 0);
  const mu = validWeeks.reduce((s, v, i) => s + v * weights[i] / wSum, 0);

  // Standard deviation of weekly performance
  const variance = validWeeks.reduce((s, v) => s + Math.pow(v - mu, 2), 0)
    / Math.max(validWeeks.length - 1, 1);
  const sigma = Math.sqrt(variance);

  // Sprint-type boost in last 2 weeks
  const weeksRemaining = Math.max(Math.ceil(remDays / 5), 0);
  const isLateperiod = weeksRemaining <= 2;
  const sprintMult = (profile?.style === 'sprint_type' && isLateperiod)
    ? profile.endPeriodMult * 0.5 : 1;

  const projectedGain = mu * sprintMult * weeksRemaining;
  const projectedTotal = current + projectedGain;
  const projectedPct = (projectedTotal / target) * 100;

  // Probability of reaching thresholds
  const gainSigma = sigma * Math.sqrt(weeksRemaining);
  const calcP = (threshPct) => {
    const needed = (threshPct / 100) * target - current;
    if (needed <= 0) return 0.99;
    return normalCDF((projectedGain - needed) / Math.max(gainSigma, 1));
  };

  return {
    projectedPct: projectedPct.toFixed(1),
    mu_weekly: mu, sigma_weekly: sigma,
    p_85: calcP(85),
    p_91: calcP(91),
    p_100: calcP(100),
    scenario_base: projectedPct,
    scenario_bull: ((current + mu * 1.15 * weeksRemaining) / target * 100),
    scenario_bear: ((current + mu * 0.75 * weeksRemaining) / target * 100)
  };
}

        'Access-Control-Allow-Origin': 'https://YOUR-USERNAME.github.io'

Performance Optimization Recommendations

Data Processing

  • IndexedDB CSV cache: Cache parsed data with a version key. On next load, compare hash of first 100 chars of CSV — if unchanged, use cache. Saves 200–400ms per page load.
  • Web Workers for AI computation: Run profiler, scorer, forecaster in a Web Worker to keep the UI thread free. The engine is currently sync and blocks rendering.
  • Incremental rendering: Large tables (eczane, brick detail) should render the top 50 rows immediately and load more on scroll using IntersectionObserver.

API Cost Control

  • Semantic deduplication: Hash the engineContext object. If the same context hash is requested within 30 minutes, serve the cached Claude response.
  • Max tokens guard: Cap all Claude calls at 800 tokens. Strategy-specific prompts should need <600 tokens. Reject any prompt exceeding the limit.
  • Batching: If manager requests analysis of all reps, batch into one multi-rep prompt rather than N individual calls.
  • Local first: Run all scoring/forecasting/RCA locally. Call Claude only if the user explicitly requests narrative. Estimate: 3–5 calls/day per rep.

Bundle Size

  • Dynamic imports: The AI engine page loads ai/ modules on demand (import()). Main app bundle stays lean.
  • Chart.js tree-shaking: Import only needed Chart.js components instead of the full 200KB bundle. Saves ~80KB.
  • Base64 audio compression: The embedded SoundFX base64 adds ~32KB to HTML. Move to a separate audio.js that is only loaded after first user interaction.

Rendering

  • Document fragment batching: Current table renders use repeated innerHTML concatenation. Switch to DocumentFragment + appendChild for tables >50 rows — 3–5× faster.
  • Chart throttling: Current chart rebuilds on every filter change. Add 150ms debounce on filter inputs to avoid redundant rebuilds.
  • CSS containment: Add contain: layout style to card components. Prevents style recalculations from propagating through the DOM on each update.
